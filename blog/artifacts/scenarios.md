# Six-scenario evidence map

This file is the provenance index for the blog's six enforcement
scenarios. Each scenario names the **real** in-code primitive it
exercises (verb / domain / runtime action — nothing invented), the
control surface that renders it, and the captured artifact(s) that
prove it. Every figure quoted in the posts traces back to one of these
files, the [efficacy report](./efficacy-report.json), or the
[edge datasheet](./edge-performance-datasheet.md).

Fleet seeded: **9 tenants across 5 compliance regimes**
(us-baseline · uk-dpa · eu-gdpr · ca-pipeda · au-privacy), **35 sites**,
**27 devices** — see [seed-summary.json](./seed-summary.json). Canonical
tenant UUIDs are pinned in `blog/harness/seed/bootstrap.go` so every
payload below is byte-for-byte reproducible across reseeds.

## Notes on honesty

- **Scenarios 1–4** are expressed as typed rules in one published policy
  graph (`PUT /api/v1/tenants/{tid}/policy`, version 2) and compiled into
  signed per-target bundles (`POST …/policy/compile`). The QoS class +
  SLA targets on the two `steer` rules ride in `Rule.Extra` and are
  re-emitted deterministically into the SD-WAN bundle slice; the class
  strings (`business-critical`, `real-time`, `best-effort`) match the
  Rust `crate::sla::SlaClass::as_str` contract.
- **Scenario 5 (throttle)** has no policy-graph verb — it is a
  data-plane decision. The artifact is generated by running the real
  `sng-swg::RateLimiter` (5-token bucket, 1/s refill): requests 1–5 are
  permitted, 6–8 are shed with `retry_after_secs = 1`, and the resulting
  `Verdict::rate_limit` maps to a 429 + `Retry-After` exactly as
  `ExtAuthzResponse::from_verdict` does (asserted by the crate's unit
  tests). Regenerate with
  `cargo run -p sng-swg --example throttle_evidence -- <out.json>`.
- **Scenario 6 (threat)** efficacy numbers are measured by the
  `sng-efficacy` harness on this VM (kernel nftables conformance + IPS
  via a real Suricata binary + an adversarial-evasion corpus). The
  `dlp_ml_ner` driver runs against ONNX Runtime 1.22. See the report
  header for the git SHA and host.
- **DLP review queue** (screenshot `scenario-04-dlp-review-queue.png`)
  has no "create event" API — in production the rows come only from the
  coach-first endpoint signal. The four pending rows in the screenshot
  are seeded directly with `blog/harness/seed/dlp_review_seed.sql`,
  mirroring the exact redacted shape the real producer writes
  (kind/label/count/severity only). Re-runnable and deterministic.
- **Per-tenant isolation is enforced, not assumed.** The capture harness
  hits per-tenant endpoints (`/api/v1/tenants/{tid}/…`) with a global-admin
  token, so the *only* thing that scopes each list to its tenant is Postgres
  row-level security. RLS is bypassed for superuser/`BYPASSRLS` roles, so the
  control plane must run as the non-superuser `sng_app` role (via
  `PG_APP_ROLE=sng_app`, adopted per-connection with `SET SESSION ROLE`).
  These artifacts were captured with that role active — which is why, e.g.,
  `s5-acme-dlp-policies.json` shows Acme's **one** PCI-DSS policy (not the
  fleet-wide list) and `s5-nordic-dlp-policies-emptystate.json` is a genuine
  empty list. Capturing as the bare superuser login silently returns
  cross-tenant rows; the harness counts therefore filter defensively with
  `rowBelongsToTenant` as a second line of defence.
- **Compliance regimes** (`pt-applied-*.json`) are the smart-default
  baselines the policy-template engine renders from each tenant's
  `(industry, country)` coordinates — one selection compiles to a
  jurisdiction-correct graph. The `pt-preview-*` request/response pairs
  show the deterministic render for finance/DE and healthcare/CA.
